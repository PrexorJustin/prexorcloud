# README template + rewrite runbook

This file is **both** a task spec for Claude Code and the canonical template every
README in this repo must match. The goal is consistency: a reader should not be
able to tell which package they're in from the *shape* of its README.

## How to invoke

Open Claude Code at the repo root and say:

> Read `README.template.md` and rewrite every README in this repo to match it.

The **Instructions** section below is the task. The **Template** section is the
structure to emit.

## About this project

> Context for Claude Code. This describes the repo these READMEs belong to so the
> rewrites land in the right voice and use the right vocabulary. **Edit this to
> match your project before running.**

**PrexorCloud is Kubernetes for Minecraft** — an open-source, MIT-licensed,
self-hosted control plane that orchestrates Minecraft servers across a fleet of
nodes. It ships as four surfaces that must feel like one product: a marketing
site + docs, a web dashboard, a CLI (`prexorctl`), and the social/README assets.

The audience is operators and SREs running game-server infrastructure, plus
developers integrating against the dashboard backend. The tone is sober infra
tooling — closer to Linear / Vercel / Tailscale than to gaming control panels.
Every README should read like one written by an SRE for other SREs.

## Instructions for Claude Code

### Scope — what to touch

1. Find every README at any depth:
   ```bash
   git ls-files | grep -iE '(^|/)readme\.mdx?$'
   ```
2. The **repo-root README** is the full template (hero banner + badges + all
   sections). **Module / sub-package READMEs** use the same section order but may
   drop the hero banner and badge row. Never drop or reorder the prose sections.
3. **Do not touch:** `README.template.md` (this file), `CHANGELOG*`, `LICENSE`,
   anything carrying a "DO NOT EDIT" / "generated by" header, or vendored /
   `node_modules` / `dist` paths.

### Process — per file, in order

1. **Read** the existing README in full. Inventory the real, accurate information
   it already holds: commands, config keys, file paths, ports, env vars, diagrams,
   links. This is your source of truth.
2. **Map** that information onto the template sections. Reshape and rewrite for
   the house voice — don't just reorder paragraphs.
3. **Preserve** every accurate technical fact: exact commands, flags, file paths,
   env var names, ports, version constraints, working links. If a command's output
   is shown, keep a real result block.
4. **Fill** placeholders from the actual package — read its `go.mod` /
   `package.json` / `Cargo.toml` / source for the real name, purpose, entry points.
   Never leave a `{{PLACEHOLDER}}` in the output.
5. **Delete** sections that genuinely don't apply (a library with nothing to
   troubleshoot drops Troubleshooting). Keep the order of the sections that remain.
6. **Do not invent.** If you can't verify a fact from the source, leave it out
   rather than guess. No aspirational features, no benchmarks you didn't find, no
   fabricated config keys. An empty section beats a wrong one.

### Accuracy guardrails

- Every command must be copy-paste runnable against **this** repo. Can't confirm
  it from the source? Don't print it.
- Use real names and values from the codebase — `node-2`, `lobby`, `:8080` — never
  `foo` / `bar` / `<your-thing-here>`.
- Keep working links; fix broken relative paths; never invent URLs.
- If the README documents something the code no longer does, drop it.

### Output

- Overwrite each README in place. One commit per file, or a single
  `docs: normalize READMEs to template`, are both fine.
- After rewriting, print a summary table: **file · sections kept · sections
  dropped · facts you could NOT verify** (so a human can confirm those).
- If a README can't be safely rewritten (missing source, ambiguous ownership),
  **skip it** and list it in the summary instead of guessing.

## House style — non-negotiable

> Mirror of `SKILL.md` "Voice & content rules" + `README.md` §5. If those files
> exist in the repo, they win on any conflict.

**Voice**

- Sound like an SRE writing for other SREs. Calm, plain, technical.
- No hype words — never "blazing-fast", "magical", "supercharged", "seamless",
  "powerful", "robust". State what it does and for whom. Receipts over adjectives.
- Marketing voice is second person ("You run…"); docs are imperative ("Run…",
  "Add…"); declarative for descriptions. Pick one per file and hold it.

**Mechanics**

- Sentence case for all headings and prose. Title Case only inside button labels.
- No emoji in prose. CLI glyphs are fine where they belong: `✓ ✗ ● ○ → ▶ │ └ ├`.
- Numbers are nouns — tabular, never abbreviated below 1k: "3 nodes", "147
  players", "1.2k crashes".
- Use the glossary terms **exactly** — capitalized, never swapped for synonyms.
  Say "Instance", not "server". "Node", not "machine". "Group", not "fleet".

  | Term | Meaning |
    |---|---|
  | **Node** | a host machine connected to the cluster |
  | **Instance** | one running Minecraft server process on a node |
  | **Group** | a template + scaling policy that spawns instances (e.g. `lobby`) |
  | **Network** | a Velocity/BungeeCord proxy fronting one or more groups |
  | **Template** | versioned filesystem + JVM args used to start instances |
  | **Catalog** | first-party + third-party server platforms (Paper, Fabric, …) |
  | **Module** | a runtime-loaded plugin extending the dashboard backend |
  | **Crash** | a non-graceful instance exit (OOM / SIGKILL / ERROR) |

- Every command is copy-paste runnable and shows its result.
- Failure / troubleshooting copy names the **fix**, not just the symptom. Not
  "Connection failed" — instead: "Can't reach node-2 on :8080 — check the firewall
  or run `prexorctl node ping node-2`."
- Link, don't repeat. Point at the deep-dive doc instead of inlining it.

**Don't**

- No filler. If a section feels empty, remove it — don't pad it.
- No marketing copy promising undocumented features.
- No emoji in the dashboard/CLI/docs voice. README badges are the only exception.
