# Contributing to PrexorCloud

Thanks for considering a contribution. PrexorCloud is an Apache 2.0,
single-maintainer project, but bug fixes, recipe additions,
documentation improvements, and well-scoped features are welcome.

If you're evaluating PrexorCloud as an operator rather than a
contributor, the [getting-started guide](docs/public/en/getting-started/what-is-prexorcloud.md)
is the better entry point.

## Prerequisites

PrexorCloud is a polyglot monorepo. To build everything locally:

- **Java 25 (preview)** — controller, daemon, modules, security; built with `--enable-preview`.
- **Java 21** — `cloud-api` (the public SPI) and in-server / proxy plugins; matches the Minecraft host runtimes.
- **Node.js 22+ with `pnpm`** (corepack-managed) — dashboard, website, and SDK packages.
- **Go 1.24+** — the `prexorctl` CLI.
- **Docker** — for the reference Compose stack and the integration test harness.
- **`buf`** — protobuf tooling for `cloud-protocol`.
- **`cosign`** (optional) — only if you want to sign module bundles locally.

The Gradle wrapper handles every Java toolchain — install at least one
JDK from each major version and Gradle will pick up the rest.

## Build everything

```bash
# Java side: controller, daemon, modules, plugins, test harness.
cd java && ./gradlew build

# Dashboard
cd ../dashboard && pnpm install && pnpm dev      # :3000

# CLI
cd ../cli && go build -o prexorctl .

# Marketing site / docs
cd ../website && pnpm install && pnpm dev        # :4321
```

The first build is slow (Gradle dependency resolution + protobuf
codegen); incremental builds are fast.

To bring up a full local cluster:

```bash
docker compose -f deploy/compose/compose.yml up -d
```

The controller logs the initial admin password on first start; pull it
out of the logs and use it with `prexorctl login`.

## Run checks before opening a PR

Three commands cover what CI runs:

```bash
# Java: formatting + full test suite
cd java && ./gradlew spotlessCheck build

# Dashboard: lint + tests
cd dashboard && pnpm lint && pnpm test

# CLI: vet + tests
cd cli && go vet ./... && go test ./...
```

For the website:

```bash
cd website && pnpm check && pnpm check:links && pnpm build
```

## Code conventions

These hold across the whole repo:

**Backend (Java)**
- **JSON: Jackson only.** No Gson, no other JSON libraries.
- **Logging: SLF4J.** No `System.out.println`. One documented exception: interactive CLI prompts in `cloud-daemon`'s `InteractiveSetup` use `System.out` because they are operator-facing terminal I/O, not log output.
- **Persistence: JDBC.** No ORMs.
- **DI: constructor injection only.** No `@Inject`, `@Autowired`, `@Component`. One documented exception: the Velocity plugin (`cloud-plugins/.../velocity/PrexorCloudVelocity.java`) uses Guice `@Inject` because Velocity's plugin loader requires it.

**Java toolchain (per-module)**

Three convention plugins live under `java/build-logic/src/main/kotlin/` — pick the right one when adding a new module:

- `prexorcloud.java21-api` → `cloud-api` (the public SPI, Java 21).
- `prexorcloud.java21-compat` → `cloud-plugins/*` (in-server plugins, Java 21 to match the host MC server runtimes).
- `prexorcloud.java25-preview` → controller, daemon, modules-core, controller modules. Java 25 with `--enable-preview`.

Don't standardise to one — see [ADR 25 in `decisions.md`](docs/engineering/decisions.md).

**Architecture**
- Respect module boundaries (controller, daemon, modules, plugins). Don't introduce cross-module shortcuts.

**Testing**
- Mock only at external boundaries (DB, network, filesystem). Internal services run real.

## PR workflow

- Branch from `main`, push your branch, open a PR against `main`.
- Use conventional-commit titles (`feat:`, `fix:`, `docs:`, `refactor:`, `chore:`, `test:`, `ci:`).
- Keep PRs scoped — one feature or fix per PR is easier to review than a sweep.
- Tests are mandatory for non-trivial code changes. Mock only at external boundaries (DB, network, filesystem); internal services run real.
- Documentation goes under `docs/public/en/` — that subtree is the source of truth for the marketing site (`website/`).
